## Compact Rules

### brainstorming
- Ask one clarifying question at a time before any implementation.
- Explore project context first, then present 2-3 approaches with tradeoffs.
- Present a design and get approval before writing code or using implementation skills.
- Save approved design to docs/plans, then hand off to writing-plans.

### writing-plans
- Write exact file paths, commands, expected outputs, and task steps.
- Break work into 2-5 minute actions with explicit TDD steps.
- Save plans in docs/plans with architecture, goal, and tech stack header.
- Keep plan DRY, YAGNI, and ready for execution without extra context.

### test-driven-development
- No production code without a failing test first.
- Follow strict red-green-refactor and verify the test fails for the right reason.
- Write minimal code to pass, then refactor while staying green.
- Treat skipping TDD as a process violation, not an optimization.

### verification-before-completion
- Never claim success without fresh verification evidence from the proving command.
- Identify the exact command, run it, read full output, then state status.
- Partial checks and assumptions do not justify completion claims.
- Use actual results, not confidence language.

### systematic-debugging
- Investigate root cause before proposing or applying fixes.
- Reproduce consistently, read the full error, check recent changes, trace data flow.
- Add instrumentation at boundaries in multi-layer systems.
- Fix the source of the bad state, not the downstream symptom.

### using-superpowers
- Invoke relevant skills before any response or action.
- Process skills come before implementation skills.
- If a skill might apply, load it instead of rationalizing.
- Follow loaded skills exactly when they impose rigid workflows.

### requesting-code-review
- Request review after major tasks or before merge.
- Compare a clear base SHA and head SHA range.
- Fix critical and important findings before proceeding.
- Push back on wrong feedback with technical reasoning.

### receiving-code-review
- Verify review comments against codebase reality before implementing.
- Clarify unclear feedback before touching code.
- Avoid performative agreement; respond technically or act directly.
- Apply changes one item at a time and test each.

### executing-plans
- Review the plan critically before executing it.
- Execute in small batches with verification after each task.
- Stop immediately on blockers or unclear instructions.
- After full verification, use finishing-a-development-branch.

### subagent-driven-development
- Use only when plan tasks are mostly independent and stay in-session.
- Fresh implementer per task, then spec review, then code-quality review.
- Do not advance until the current task passes both review gates.
- Finish with a branch-finishing workflow.

### branch-pr
- Inspect status, diff, commits, and base-branch delta before opening a PR.
- Summarize the whole branch, not just the last commit.
- Push with upstream tracking if needed, then create PR with structured summary.
- Return the PR URL.

### issue-creation
- Create issues with problem, context, scope, and acceptance details.
- Use GitHub tooling, not ad-hoc text outside the tracker.
- Be explicit about why the issue matters and how success is measured.

### judgment-day
- Run dual blind review passes against the same target.
- Synthesize findings, fix issues, then re-judge up to the allowed loop.
- Escalate after repeated failure instead of hand-waving readiness.

### skill-creator
- Define clear triggers, constraints, and workflows for new skills.
- Document only reusable patterns worth operationalizing.
- Keep skills actionable and easy for agents to apply consistently.

### writing-skills
- Update skill docs carefully and verify trigger clarity.
- Prefer concise, high-signal instructions over verbose tutorials.
- Validate that the skill is safe, coherent, and operational.

### using-git-worktrees
- Create isolated worktrees before risky or parallel feature work.
- Verify branch/worktree state before starting implementation.
- Keep the current workspace clean and avoid cross-branch contamination.

### finishing-a-development-branch
- Confirm tests and verification pass before proposing merge options.
- Present integration choices with tradeoffs instead of assuming one path.
- Clean up branch workflow only after user-approved direction.

### dispatching-parallel-agents
- Use parallelization only for truly independent tasks.
- Split work to minimize shared state and synchronization overhead.
- Recombine outputs with explicit verification.

### go-testing
- Prefer real Go testing patterns and Bubbletea-specific test tooling.
- Keep tests focused on behavior and terminal interactions.
- Use idiomatic Go assertions and deterministic fixtures.

### astro
- Prefer src/pages for routes, src/layouts for document shells, and public for static assets.
- Use Astro for content structure first; hydrate React only where interaction is needed.
- Re-run Astro type/sync checks after integration changes when verification allows.
- Keep site URL and output mode aligned with deployment intent.

### astro-ui-patterns
- Use transition:name for morphing elements across pages.
- Use transition:persist for stateful persistent UI between navigations.
- Default to deferred hydration with client:idle or client:visible; reserve client:load for critical interactivity.
- Prefer CSS/Tailwind animations over JS for static surfaces.

### shadcn
- Use existing shadcn components and variants before custom markup.
- Keep className for layout/composition, not component recoloring.
- Prefer semantic tokens, gap utilities, cn(), and required accessibility subcomponents.
- For dialogs/drawers/sheets, always include titles and proper composition.

### tailwind-v4-shadcn
- Use @theme inline and CSS variables as the source of truth for tokens.
- Avoid old Tailwind v3 patterns when working in v4 projects.
- Keep theming semantic and compatible with shadcn primitives.
- Check known gotchas before “fixing” color or dark-mode issues ad hoc.

### tailwind-css-patterns
- Start mobile-first and layer responsive prefixes upward.
- Compose utilities from tokens instead of ad-hoc CSS values.
- Extract repeated patterns thoughtfully and keep layouts accessible.
- Verify behavior at breakpoints and avoid unnecessary custom CSS.

### frontend-design
- Commit to a clear visual direction before coding.
- Avoid generic AI-looking typography, palettes, and layouts.
- Match implementation complexity to the aesthetic goal.
- Optimize for distinctive, production-grade interfaces.

### make-interfaces-feel-better
- Use concentric radii, optical alignment, and layered shadows for polish.
- Prefer interruptible transitions and exact transition properties.
- Use subtle exit motion, tactile scale-on-press, and text-wrap/font smoothing where relevant.
- Apply will-change sparingly and only on compositor-friendly properties.

### emil-design-eng
- Animate transforms and opacity, not layout properties.
- Use origin-aware motion and non-default easings.
- Start scale entrances from visible values, never 0.
- Respect prefers-reduced-motion with simplified effects.

### vaul-aware-animations
- Enable shouldScaleBackground on the Vaul root for aware animations.
- Body must provide the dark void; app content must live inside [vaul-drawer-wrapper].
- Give the wrapper the real app background and min-h-screen.
- Keep DOM structure aligned with Vaul’s expected scaling behavior.

### vaul-drawer-polish
- Do not fight Vaul physics with animate-in, slide-in, or transition-all on DrawerContent.
- DrawerContent must be the visible surface, not a transparent wrapper around one.
- Use tactile scale, transition-transform, 40x40 hit targets, and concentric radii inside drawers.
- Prefer soft shadows over hard borders.

### gsap-core
- Prefer GSAP for general-purpose JS animation when no library is chosen.
- Use x/y/scale/rotation aliases and camelCase CSS properties.
- Reach for to/from/fromTo/set appropriately and choose intentional easing.
- Use matchMedia and reduced-motion aware patterns where needed.

### gsap-scrolltrigger
- Register ScrollTrigger once before use.
- Use start/end/trigger intentionally; remove markers in production.
- Pin containers thoughtfully and animate children rather than pinned shells.
- Refresh only when layout actually changes.

### gsap-timeline
- Use timelines for multi-step or coordinated animation.
- Control sequencing with labels and position parameters instead of delay soup.
- Set timeline defaults to reduce repetition and improve consistency.

### gsap-utils
- Use utility helpers for clamping, mapping, snapping, interpolation, and normalization.
- Prefer function form when reusing the same transform repeatedly.
- Keep animation math declarative instead of hardcoding repeated calculations.

### gsap-performance
- Prefer transform and opacity to stay on the compositor.
- Use will-change sparingly and only on animated elements.
- Batch reads/writes, use stagger/quickTo for repeated updates, and clean up inactive animations.
- Avoid heavy simultaneous tweens and layout-thrashing properties.

### gsap-plugins
- Register every GSAP plugin once before use.
- Use the right plugin for scrollTo, FLIP, dragging, SVG, text, or easing work.
- Keep plugin setup outside re-render loops.
- Follow plugin-specific DOM structure requirements.

### accessibility
- Follow WCAG 2.2 AA as the default target.
- Ensure alt text, accessible names, keyboard access, and semantic structure.
- Treat decorative media as presentational and interactive controls as labeled.
- Validate operable, understandable, and robust behavior, not just visuals.

### seo
- Set crawlability, canonical URLs, and metadata intentionally.
- Avoid duplicate-content issues and keep pages indexable unless intentionally excluded.
- Use structured data and sitemap/robots support when relevant.
- Treat technical SEO as a first-class quality concern.

### nodejs-best-practices
- Choose frameworks based on deployment target, performance needs, and team context.
- Prefer principle-driven decisions over copying one default stack everywhere.
- Consider runtime, async, and security implications up front.
- Ask for context when tradeoffs are unclear.

### nodejs-backend-patterns
- Build backends with explicit middleware, error handling, auth, and logging.
- Separate concerns cleanly across routing, services, and persistence.
- Design for production readiness, not tutorial minimalism.
- Prefer scalable patterns that survive growth.

### nodejs-express-server
- Use structured middleware chains, health routes, and centralized error handling.
- Keep request parsing, auth, and routing organized and explicit.
- Build Express apps like services, not ad-hoc scripts.

### typescript-advanced-types
- Reach for generics, mapped types, conditional types, and utility types deliberately.
- Use types to model invariants and API guarantees, not just satisfy the compiler.
- Prefer reusable type abstractions over repeated manual unions.

### vercel-composition-patterns
- Avoid boolean-prop proliferation; create explicit variants and compound components.
- Lift state into providers and expose generic context interfaces.
- Prefer children composition over render-prop customization unless data injection is required.
- In React 19, prefer use() and regular ref props over older patterns where appropriate.

### notebooklm-py
- Treat the API as unofficial and potentially unstable.
- Use async workflows, wait for generation completion, and require login setup.
- Default generated/exported content to Spanish when using this toolchain.

### obsidian-cli
- Use the obsidian CLI against a running Obsidian instance.
- Prefer exact path/file targeting and quote values with spaces.
- Reload plugins and inspect errors/screenshots during plugin development cycles.

### obsidian-markdown
- Use frontmatter, wikilinks, embeds, and callouts with valid Obsidian syntax.
- Prefer wikilinks for vault-local notes and Markdown links for external URLs.
- Validate render behavior in Obsidian reading view.

### obsidian-bases
- Create valid YAML .base files with filters, formulas, views, and summaries.
- Validate quoting and formula references carefully.
- Test the rendered base in Obsidian after editing.
